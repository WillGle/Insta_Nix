# Documentation Style Guide

## Purpose

This file defines the writing style for documentation in this repository.

## When to use

Use this guide when adding a new document or rewriting an existing one.

## Prerequisites

- Know whether the file is an active guide or archived material.
- Preserve exact commands, paths, host names, log strings, and dates when they matter technically.

## Steps

1. Write short sentences.
2. Prefer direct nouns such as "host", "file", "required labels", and "common commands".
3. Avoid internal framework language unless it is technically required.
4. Keep introductions short. Put commands and checks near the top.
5. Link to related documents instead of repeating long explanations.

## Verification

Active guides should use these headings:

- `Purpose`
- `When to use`
- `Prerequisites`
- `Steps`
- `Verification`
- `Related docs`

Archive docs should start with these headings:

- `Status`
- `Summary`
- `Historical details`
- `References`

Avoid these words in active guides unless they are part of a literal path, command, or proper noun:

- `taxonomy`
- `contract`
- `command matrix`
- `output surface`
- `guardrail`
- `governance`
- `mission boundaries`
- `execution protocol`
